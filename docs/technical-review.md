## Confirmed findings

### High severity

- **[ai-engineering] Judge/eval 'temperature 0 determinism' claim is false for every model in the allowlist** — `lib/eval/runner.ts:29`
  Claim is accurate as written. Two minor refinements: (1) the ops-level OPENROUTER_MODEL env override could route to a temperature-supporting model, so the no-op is only guaranteed for the three UI-offered models; (2) OpenRouter's GPT-5 models DO support the 'seed' parameter, which the app never sends — an available (best-effort) stability mitigation that would partially honor the determinism intent, alongside options like pinning reasoning effort or switching the judge to a temperature-supporting model.
  _Fix:_ Pin the judge/labeler to a model that actually honors temperature (or verify supported_parameters via OpenRouter's /models endpoint at startup), pass a seed where supported, and soften every 'deterministic' claim to 'judge config held fixed'. For real grade stability run the judge k times and take majority verdicts. Either remove the temperature slider for GPT-5-family models or surface that it has no effect.

- **[ai-engineering] Gold set is saturated: distractors are strawmen with zero discriminative power, so the headline metrics are always ~100%** — `lib/eval/goldset.ts:93`
  The negative class of the gold set (lib/eval/goldset.ts) has zero observed discriminative power: most distractors are claims no competent model would assert (verified verbatim at lines 58, 94, 95, 129), and the project's own recorded results (docs/prompt-strategies.md tables, lines 53-77) show 0 asserted distractors across all 10 strategy-by-model cells (FPR 0% everywhere, precision 100% everywhere), with every strategy tied at 100% P/R/F1 on the default model (gpt-5-mini). Corrections to the original claim: (1) the instrument is NOT fully dead — recall retains dynamic range on weaker models (gpt-5-nano: 92.6-96.3% recall in 4/5 strategies), so missed-key-point regressions would be detected; what cannot be detected is false-claim regressions (the precision/FPR side) and any strategy separation at the default operating point. (2) The DEFAULT_STRATEGY=zero-shot decision (strategies.ts:124-130) additionally used latency/cost and the secondary unsupported-claims counts as tiebreakers, which did differ; only the headline quality metrics 'showed nothing'. (3) The saturation is self-documented — 'The gold set is saturated... The metric can't separate them' is at docs/prompt-strategies.md:61-62 (not 79-84), and the Caveats section (lines 107-117) explicitly recommends 'trickier distractors that actually catch models out' — so this is a disclosed limitation of a small hand-built gold set rather than a hidden defect, though README lines 4-6 and the strategies.ts:6 'real measurement instrument' comment present the harness without that caveat. The saved-interview regrade path (lib/eval/autolabel.ts) derives fresh labels per question and is unaffected. Item/label counts confirmed exact: 7 items, 27 keyPoints, 21 distractors.
  _Fix:_ Add harder, senior-level items and near-miss distractors that capable models actually assert (e.g. 'the virtual DOM makes React faster than manual DOM manipulation in all cases', 'useEffect cleanup runs only on unmount', 'microtasks can be preempted by rendering'). Grow the set until baseline scores land in a 60-90% band where differences are measurable.

- **[correctness] Streaming path drops the empty-completion retry, then misattributes the failure to token exhaustion** — `lib/openrouter.ts:320`
  The claim is accurate as written. One refinement: the empty-stream error state is not purely a misattribution edge case — because streamChat sends maxTokens + 2048 reasoning-headroom tokens upstream (min effective 2304), genuine reasoning-starvation empties are largely engineered away, so a zero-delta stream is dominated by the sporadic-empty failure mode the non-streaming path retries and the code comments call common. The client message ('Increase Max tokens') is therefore wrong advice in the dominant case, and the [DONE]-with-no-deltas, immediate-close, and mid-stream data:{"error":...} cases (silently skipped at lib/openrouter.ts:288 since the event has no `choices`) all funnel into it via InterviewForm.tsx:113-120. The same no-retry gap also applies to the follow-up answer stream (app/api/answer/route.ts → streamAnswer), though there the client only shows a generic empty/error state.
  _Fix:_ In streamChat, buffer until the first non-empty delta; if the stream ends with zero content yielded, retry the connection (reusing the connectChatStream backoff) up to MAX_ATTEMPTS before committing bytes to the caller. Also surface mid-stream `error` events from the SSE payload. On the client, use a generic error message when the raw stream is empty rather than asserting token exhaustion.

- **[security] Rate limiting keyed on client-controlled X-Forwarded-For — spoofable bypass and global-lockout failure mode** — `lib/security.ts:440`
  Core claim stands. Corrections: (1) The gold set now contains 8 items (react-vdom, react-useeffect, ts-unknown-any, js-event-loop, css-flex-grid, next-server-components, sysdesign-debounce-throttle, plus one more), so one POST /api/evaluate fans out ~16 OpenRouter calls (8 items x generate+judge), not 14 — the abuse amplification is slightly larger than claimed, not smaller. (2) Scenario B (global lockout via shared "unknown" bucket) is real but should be qualified: Next.js 16 backfills x-forwarded-for from socket.remoteAddress when it is absent AND the request is a NodeNextRequest (base-server.js:577), so a directly-exposed Node server would give distinct clients distinct keys rather than a single "unknown" bucket; the full-collapse-to-'unknown' failure mode primarily applies in edge/minimal-mode runtimes or deployments where proxy headers are stripped and remoteAddress is unavailable. Scenario A (spoof/rotate leftmost XFF for a fresh bucket every request) is unconditional and is the more severe of the two. Recommended fix: do not trust client-supplied XFF for the rate-limit key; use a platform-provided trusted client IP (e.g. a configured trusted-proxy hop count / the rightmost trusted entry) or gate the paid endpoints behind auth, and update the README to state the limiter only meaningfully keys on IP behind a trusted proxy.
  _Fix:_ Trust proxy headers only when a trusted proxy is known to set them (e.g., an env flag for the deployment platform); when parsing XFF behind a known chain, take the rightmost non-trusted hop, not the leftmost entry. For direct exposure, add an auth token / origin check or a global (not per-key) request budget as backstop, and say so in the README.

### Medium severity

- **[ai-engineering] Chain-of-thought strategy is CoT in name only: reasoning is explicitly forbidden from the output** — `lib/prompts/strategies.ts:68`
  Substantively correct with three refinements. (1) The title overstates: reasoning is EXPLICITLY forbidden only in the interview-generation variant (lib/prompts.ts:51, 'never the intermediate reasoning'); in the eval strategy (strategies.ts:68-73) the suppression is implicit ('before writing' + short-answer contract), and the author knowingly documented the design — the code comment (strategies.ts:65-67) argues 'the reasoning becomes the answer' since the answer must 'walk through the mechanism in order', and README line 95 states '(but only the final answer is emitted)'. So some ordered reasoning IS emitted inside the answer; the valid core of the claim is that zero-shot answers to explain-style questions produce the same artifact, so the differential emitted compute between arms is ~zero. (2) The 'matched the baseline is exactly what you'd expect from an unimplemented technique' argument is weak evidence: the gold set is saturated on gpt-5-mini (all five strategies scored 100%), so even a faithful CoT could not have separated from baseline there. (3) Practical impact is bounded: the doc's decision to default to zero-shot rests on saturation plus latency and would be unchanged by a faithful CoT arm — the flaw is in the validity of the labeled comparison (a doc/benchmark-integrity issue), not in app behavior. Keep severity medium.
  _Fix:_ Implement CoT faithfully: let the model emit its reasoning followed by a delimiter (e.g. 'FINAL ANSWER:'), grade only the extracted answer — the output contract for the judge stays intact. Alternatively, reframe the comparison as reasoning-effort levels (minimal/low/medium/high), which is the real CoT knob on these models, and rename the current prompt to something honest like 'mechanism-walkthrough'.

- **[ai-engineering] 'Precision'/F1 mix two disjoint label universes and ignore all claims outside them; the UI tooltip mis-describes the metric** — `lib/eval/metrics.ts:72`
  The metric design is a deliberate, documented closed-universe rubric eval, not an incoherent hybrid: precision/recall/FPR are standard binary-classification metrics over the labeled claim universe (keyPoints = positive class, distractors = negative class), as documented in types/eval.ts:6-9 and README. Claims outside both lists are captured as unsupportedClaims and intentionally kept out of the matrix, with that exclusion disclosed in types/eval.ts:84-86, docs/prompt-strategies.md:36-37, and directly in the UI ('Unsupported claims (not in matrix)', EvalDashboard.tsx:887) — and the docs use the unsup count as a secondary quality signal when choosing the default strategy. The one real defect is narrow: the precision tooltip at components/EvalDashboard.tsx:59-60 opens with 'Of all the factual claims in the answer...', which misdescribes the denominator (only labeled claims count; fabricated off-list claims do not lower precision), and 'fewer incorrect statements slip in' overstates for the same reason. Fix is a one-sentence copy edit, e.g. 'Of the labeled claims the answer asserted, the share that were required key points rather than known-wrong distractors; wrong claims outside the labeled set are tracked separately as unsupported claims.' Severity: low.
  _Fix:_ Report the two things actually measured under honest names: key-point recall (coverage) and distractor assertion rate (known-misconception rate). If a precision-like number is wanted, have the judge classify every claim in the answer (supported/unsupported) and compute precision over that, folding unsupportedClaims into the numerator/denominator. Fix the tooltip text either way.

- **[ai-engineering] Saved-answer grading is circular: the same model family generates the answer, derives the rubric, and judges** — `lib/eval/autolabel.ts:21`
  The claim is essentially correct with two minor refinements. (1) The repo is not entirely silent on judge-model bias: docs/prompt-strategies.md has a Caveats bullet ("Judge is gpt-5-mini... A different judge could shift borderline calls"), but it covers only the judge role, only for the gold-set strategy comparison, frames the issue as borderline noise rather than systematic optimism, and says nothing about the labeler-derived-rubric circularity — the README itself has no caveat at all. (2) The labeler samples at temperature 0 while answers are generated at ~0.7, so the rubric and answer are distribution-correlated (same weights) rather than literally identically sampled; the recall-inflation conclusion is unaffected. Additionally, the model allowlist makes the circularity structural: users cannot select a non-gpt-5 model, and the OPENROUTER_MODEL ops override changes generator, labeler, and judge together.
  _Fix:_ Use a different model family for the labeler and/or judge (OpenRouter makes this a one-line change), or at minimum document the circularity next to the no-leakage claim. An ensemble judge (2 families, disagreements flagged) would be the serious fix.

- **[ai-engineering] Statistical rigor: single-run cells, n=7 items, conclusions drawn from 1-2 key-point deltas, no uncertainty quantification** — `docs/prompt-strategies.md:79`
  The claim is accurate with two corrections. (1) Not both 'robust observations' are noise: observation #1 (perfect precision/FPR everywhere) reflects 0 false positives across all 10 cells (~210 distractor judgments) and is consistent, though it mainly indicates easy distractors as the doc admits; only observation #2 (self-critique > others ranking on gpt-5-nano) rests on statistically indistinguishable 1-2 key-point deltas. (2) The shipped DEFAULT_STRATEGY='zero-shot' decision does not depend on the noisy ranking — it rests on the saturated 100% tie on gpt-5-mini plus zero-shot being deterministically cheapest in prompt tokens; the statistically unsupported part is the doc's corollary ('self-critique is the strategy to reach for on a weaker model') and the single-run latency ordering used as a tiebreaker. Additional supporting detail: persona and self-critique are currently commented out of PROMPT_STRATEGIES (lib/prompts/strategies.ts:17-23), so the dashboard can only compare 3 of the 5 documented strategies, and the doc's caveat 'treat ±1 key point as noise' (line 112) directly contradicts the body calling a ±1 delta a robust win (line 85).

- **[ai-engineering] Streaming path silently swallows mid-stream errors and truncation; incomplete answers are saved and later graded** — `lib/openrouter.ts:326`
  Claim is accurate as written; two refinements: (1) an additional, more common trigger is the interview route's 45-second hard timeout (AbortSignal.timeout(45_000), app/api/interview/route.ts:13,50-53) — its mid-stream AbortError lands in streamChat's silent catch, so slow generations are truncated and auto-saved with no signal; (2) the streamChat comment 'we cannot signal an error' overstates the constraint — controller.error() would reject the client's read() and be caught by InterviewForm, and finish_reason:"length" is detectable in-band in sseDeltas, so both detection and signaling are feasible fixes. - **[architecture] Four API routes duplicate their entire scaffolding, and error-mapping policy has already begun to drift** — `app/api/interview/route.ts:15`
  Four API routes do hand-copy identical scaffolding (json<T>() helper, rate-limit-429 block, parse-400 block, catch-mapping), and the two route pairs check different abort-error names — but that difference is deliberate mechanism-matching, not drift: the eval routes use setTimeout+AbortController.abort(), which can only produce "AbortError", so a TimeoutError check there would be dead code. The hypothesized bug ("harmonizing onto AbortSignal.timeout turns timeouts into 500s") does not hold: lib/openrouter.ts's retry loop re-normalizes aborted-signal failures into an Error named "AbortError" (delay()/signalAbort(), lines 94-113), so a TimeoutError-driven abort still maps to 504 in practically all paths (worst edge: 502; the per-item catches in lib/eval/runner.ts and savedRunner.ts would otherwise degrade it to a 200 with error rows — never a 500). ## Refuted claims

_Flagged by a reviewer, then knocked down by the adversarial verifier — evidence the confirmation process was doing real work._

- **Stale stream chunk can repaint old interview content after a new submit resets state** (correctness) — The claim fails on both of its premises. (1) The trigger cannot occur: handleSubmit sets status to "loading" synchronously before any await (InterviewForm.tsx:62), and the submit button is disabled={busy} (line 241) with busy covering loading and streaming (line 136). React flushes discrete-event state updates synchronously before the next event task, so the second click of a double-click sees a disabled button; implicit Enter submission is also blocked by the HTML spec when the default submit button is disabled. handleSubmit can therefore only be re-entered from idle/done/error, states where the prior run has fully returned and no reader continuation exists. (2) Even ignoring the guard, the claimed interleaving violates event-loop ordering: an await reader.read() continuation is a microtask, and microtasks fully drain before the next macrotask (the second submit's DOM event). If the old read had resolved, its `raw += ...; setStreamText(raw)` continuation ran BEFORE the new submit's setStreamText(""), then parked on the next reader.read() — which is precisely the pending read that abort() rejects, and that AbortError is swallowed at line 129 with no state writes. No stale setStreamText can land after the reset, so old questions can never repaint under the new run's header. The reviewer's parenthetical ("the abort only rejects the NEXT read") concedes the point: that next read is the only one that can be outstanding across the reset, and it rejects cleanly. Nothing Next.js 16 / React 19-specific changes this; it is plain fetch/streams/microtask semantics.
- **Header logo is a square image rendered at 45x20 — visibly distorted** (frontend-ux) — Empirically refuted in a live browser: the logo renders at 45x45 px, square and undistorted. While logo.png is indeed 1254x1254 and header.tsx passes width={45} height={20}, Tailwind v4 preflight (included via @import "tailwindcss" in app/globals.css) applies `img { max-width: 100%; height: auto; }`, and Next 16's Image emits width/height only as HTML attributes for non-fill images (get-img-props.js:503-515 — inline style is just color:transparent). Author CSS overrides the height attribute, so the browser computes height from the intrinsic 1:1 ratio. Playwright measurement against the running dev server confirms: rect 45x45, computed height 45px, optimizer-served bitmap square (48x48). The dev warning the reviewer cited ("width or height modified, but not the other") fires precisely because CSS overrode the height attribute — it is evidence against distortion, not for it. No visible distortion ships; the claim's core assertion and its medium severity are wrong.
